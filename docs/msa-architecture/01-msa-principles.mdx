| 원칙                                         | 세부 설명                                                    | 시나리오 예시 (온라인 쇼핑몰)                                                                   |
| -------------------------------------------- | ------------------------------------------------------------ | ----------------------------------------------------------------------------------------------- |
| **1. 단일 책임**<br/>(Single Responsibility) | 서비스는 하나의 비즈니스 기능만 담당해 응집도 ↑ 결합도 ↓     | **상품 서비스**: 상품 CRUD<br/>**주문 서비스**: 주문·결제<br/>**회원 서비스**: 로그인·정보 수정 |
| **2. 독립 배포**                             | 서비스마다 따로 빌드·테스트·배포 → 배포 주기 단축            | 상품 추천 알고리즘 교체 시 **상품 서비스**만 재배포                                             |
| **3. 분산 데이터**                           | 각 서비스가 **자신 DB** 소유, 외부는 API로만 접근            | 주문 서비스가 재고를 수정하려면 **상품 API** 호출                                               |
| **4. API 기반 통신**                         | REST / gRPC / 메시지 큐 등 표준화된 인터페이스               | 주문 완료 → Kafka 메시지 발행 → 알림 서비스가 SMS 전송                                          |
| **5. 장애 격리**                             | Circuit Breaker·Bulkhead 로 한 서비스 장애가 확산되지 않도록 | 추천 서비스 다운 → 추천 영역만 비활성, 주문은 정상                                              |
| **6. 기술 다양성**                           | 서비스 특성에 맞는 언어·DB·프레임워크 선택                   | 알림 서비스=Go, 주문 서비스=Java, 추천 서비스=Python                                            |
